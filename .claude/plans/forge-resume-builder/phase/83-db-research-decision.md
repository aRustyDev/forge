# Phase 83: Database Research — Decision & Migration Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans

**Depends on:** Phase 82 (deployment validation and ADRs)
**Parallel with:** Nothing — this is a decision gate
**Duration:** Small (4 tasks: T83.1 through T83.4, decision + planning)

## Goals

- Make a final, evidence-based database decision (including "stay on SQLite" as a valid outcome)
- If migrating: produce a detailed migration plan with phases, rollback strategy, and acceptance criteria
- If staying on SQLite: document what supplementary tools/extensions address the graph and vector gaps
- Update the Forge implementation plan (PLAN.md) with the decision and any new phases

## Non-Goals

- Executing the migration (that's a future set of phases generated by this decision)
- Changing any production code in this phase
- Choosing a deployment platform (Cloudflare decision is separate, though DB choice constrains it)

## Context

Phases 80-82 produced: a requirements audit, an evaluation rubric, deep-dive evaluations of each candidate, deployment prototypes, sync architecture designs, migration effort estimates, and Architecture Decision Records. This phase synthesizes all of that into a decision.

The decision is not "which database is best in general" but "which database is best for Forge's specific workload, deployment target, and development velocity." Staying on SQLite with targeted extensions (e.g., sqlite-vec for vectors, recursive CTEs for graph) is a legitimate and possibly superior outcome.

## Tasks

### T83.1: Decision Synthesis

**Goal:** Weigh all evidence and make the decision.

**Steps:**
1. Review the comparison matrix from Phase 81 (T81.6)
2. Review the ADRs from Phase 82 (T82.5)
3. Apply Forge-specific weighting:
   - How much do graph queries actually matter today vs. future? (Is the derivation chain traversal really that painful in recursive CTEs?)
   - How much do vectors matter? (Phase 69 already works with SQLite — is the current approach insufficient?)
   - How critical is edge deployment? (Is Forge even deploying to Cloudflare soon, or is this a future concern?)
4. Consider migration risk vs. benefit:
   - What's the blast radius of a DB swap? (Every repository, every migration, every test)
   - What's the opportunity cost of migration time vs. building features?
   - Is the current pain high enough to justify the migration effort?
5. Write the decision with explicit rationale

**Deliverable:** `refs/research/db-research/83-decision.md`

### T83.2: Migration Plan (if migrating)

**Goal:** If the decision is to migrate, produce a phased migration plan.

**Steps:**
1. Define migration phases:
   - Phase A: Create adapter/abstraction layer over repositories (allows gradual swap)
   - Phase B: Migrate schema to new DB (DDL translation, migration runner adaptation)
   - Phase C: Migrate data (export SQLite → import new DB, verify row counts and integrity)
   - Phase D: Swap repositories one-by-one behind the adapter
   - Phase E: Remove SQLite dependencies, clean up adapter if no longer needed
   - Phase F: Deploy to target environment (Cloudflare or other)
2. For each phase:
   - Estimated scope (files changed, tests affected)
   - Rollback strategy (can we revert to SQLite at this point?)
   - Acceptance criteria (what must pass before proceeding?)
3. Define the point of no return (after which rollback to SQLite is impractical)
4. Estimate total migration effort in terms of phase count (not time)

**Deliverable:** `refs/research/db-research/83-migration-plan.md`

### T83.3: SQLite Enhancement Plan (if staying)

**Goal:** If the decision is to stay on SQLite, document how to address the graph and vector gaps.

**Steps:**
1. **Graph queries:** Document how to improve derivation chain queries with:
   - Better recursive CTEs (precomputed closure table? materialized paths?)
   - Application-level graph traversal (load relevant nodes, traverse in TypeScript)
   - SQLite's JSON functions for lightweight graph-like structures
2. **Vector search:** Evaluate SQLite vector extensions:
   - `sqlite-vec` (Alex Garcia's extension — native vector search for SQLite)
   - `sqlite-vss` (older, based on Faiss)
   - Continue using application-level cosine similarity (current approach)
3. **Edge deployment:** SQLite is already the best-supported edge DB (Cloudflare D1, Durable Objects, Turso embedded replicas). Document the simplest path.
4. Produce a plan for targeted improvements that address the gaps without a full DB swap

**Deliverable:** `refs/research/db-research/83-sqlite-enhancement-plan.md`

### T83.4: Update PLAN.md and Create Follow-Up Phases

**Goal:** Integrate the decision into Forge's implementation plan.

**Steps:**
1. Add a "Database Decision" section to PLAN.md documenting the outcome
2. If migrating: create new phase files (84+) for each migration phase from T83.2
3. If staying on SQLite: create phase files for targeted enhancements from T83.3
4. Update dependency graph in PLAN.md
5. Create beads issues for the new phases

**Deliverable:** Updated `PLAN.md` + new phase files
