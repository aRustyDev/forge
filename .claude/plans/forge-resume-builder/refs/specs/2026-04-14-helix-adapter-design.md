# HelixDB Adapter + Two-Way Migration — Design Document

**Status:** Approved
**Created:** 2026-04-14
**Phase:** Storage Abstraction Phase 2
**Prerequisite:** Phase 0 (adapter interface) + Phase 1 (SQLite adapter, ELM rewiring) — both merged to `main`

---

## 1. Motivation

Phase 1 established the EntityLifecycleManager (ELM) as the single data access layer for all 26 Forge services. All 47 entities flow through the `StorageAdapter` interface. The raw repository layer is deleted.

Phase 2 proves the abstraction works by implementing a second adapter — HelixDB, a Rust-based graph+vector database. This validates that the adapter interface can absorb fundamentally different storage semantics (graph traversals, native HNSW vectors, compile-time query validation) without touching application logic.

---

## 2. Decision Record: Why Full Codegen Adapter

### The Constraint

HelixDB compiles HQL queries to Rust at build time. The TypeScript SDK (`helix-ts`) has one method: `client.query(queryName, params)`. You cannot construct queries at runtime. Every query the adapter will ever call must be pre-defined in `.hx` files and compiled before deployment.

The `StorageAdapter` interface expects generic runtime operations (`create("bullets", {content: "..."})`). This is a fundamental impedance mismatch.

### Three Approaches Evaluated

#### Approach A: Full Codegen Adapter (Chosen)

Generate every HQL query from the entity map. The adapter is a thin dispatcher that maps `(operation, entityType)` to `client.query(queryName, params)`.

- ~343 pre-compiled queries generated mechanically from the entity map
- Schema files (`src/schemas/*.hx`) hand-maintained
- Query files (`src/queries/*.hx`) machine-generated by `codegen.ts`
- Both compiled to `dist/schema.hx` + `dist/queries.hx` for HelixDB consumption
- Named queries (traceChain, etc.) hand-written to leverage graph traversals

**Pros:**
- Fully mechanical, self-maintaining — add an entity to the entity map, re-run codegen
- No manual query writing for CRUD operations
- Named queries leverage HelixDB's native graph traversals (the whole point)
- WhereClause operator gaps are solvable (audit shows only 6 operator types used)

**Cons:**
- Large generated query file (~2K+ lines) — mitigated by multi-file src/ structure
- WhereClause→HQL translation has a Tier 3 fallback to client-side filtering (~1% of calls)

#### Approach B: Minimal Codegen + Heavy Named Queries (Rejected)

Generate only basic CRUD (Add, Get, Update, Delete) per entity. Everything involving filters goes through named queries or client-side filtering.

- Fewer generated queries (~188)
- Every new filter pattern needs a hand-written named query
- Defeats the "adapter is mechanical" principle
- High ongoing maintenance burden

**Rejected because:** Every new service filter pattern would require a corresponding hand-written HQL query. The adapter would become a growing maintenance liability rather than a mechanical translation layer.

#### Approach C: HQL Template Engine (Rejected)

Build a template engine that emits HQL query strings with slots for where clauses, ordering, pagination. Pass filter expressions as string parameters.

- Smallest query surface
- Most flexible

**Rejected because:** HQL is compile-time — queries are compiled to Rust. String interpolation and runtime query construction are fundamentally incompatible with HelixDB's architecture. This approach would require changes to HelixDB itself.

### Why Approach A Wins

1. **Alignment with HelixDB's philosophy:** Compile-time query validation is a feature, not a limitation. Pre-generating all queries lets HelixDB validate the entire query surface at build time.
2. **Mechanical correctness:** The codegen reads the same entity map that the ELM uses. If the entity map is correct (validated by PRAGMA tests), the generated queries are correct by construction.
3. **Self-maintaining:** Schema changes propagate automatically: update entity map → re-run codegen → rebuild HelixDB.
4. **Named queries leverage graph:** The hand-written named queries (traceChain, listBulletsFiltered, etc.) use native graph traversals — multi-hop paths that are 3-table JOINs in SQLite become `::Out<Edge>::In<Edge>` chains in HQL.

## 3. HQL Schema Mapping

### Entity Classification

Derived at adapter startup from the entity map using this heuristic:

| Category | Count | HQL Type | Heuristic |
|---|---|---|---|
| Core entities | 30 | `N::` (Node) | Has `id` field, no composite `primaryKey` |
| Junction tables | 13 | `E::` (Edge) | Has composite `primaryKey` with 2+ FK fields |
| Embeddable entities | 4 | Also `V::` (Vector) | Entity types in `EMBEDDING_ENTITY_TYPES`: bullet, perspective, jd_requirement, source |

The adapter scans the entity map once at construction time and builds a `Set<string>` of edge entity types. This is derived, not hardcoded — adding a junction table to the entity map automatically classifies it as an edge.

### Type Mapping

| Entity Map Type | HQL Type | Notes |
|---|---|---|
| `text` | `String` | |
| `integer` | `I64` | |
| `real` | `F64` | |
| `integer` + `boolean: true` | `Boolean` | HQL has native booleans |
| `json` | `String` | Serialized to JSON string, same as SQLite adapter |
| `blob` | `String` | Base64-encode for transport (or use V:: native storage) |

### Constraint Mapping

| SQLite Constraint | HQL Equivalent | Handling |
|---|---|---|
| `PRIMARY KEY` (id) | Implicit `ID` on N::/E::/V:: | HelixDB auto-generates; our `id` stored as `UNIQUE INDEX` field |
| `UNIQUE` | `UNIQUE INDEX field: Type` | On nodes only (HelixDB limitation) |
| `NOT NULL` / `required` | No HQL equivalent | Enforced by ELM |
| `CHECK` (enum values) | No HQL equivalent | Enforced by ELM |
| `FOREIGN KEY` | Edge relationships (E::) | FK integrity enforced by ELM |
| `ON DELETE CASCADE` | No HQL equivalent | Enforced by ELM (`nativeCascade: false`) |
| `DEFAULT` | `DEFAULT value` / `DEFAULT NOW` | Simple defaults in HQL; function defaults in ELM |

### ID Strategy

HelixDB auto-generates its own internal `ID` for every node/edge/vector. Forge's UUIDs are preserved as a `UNIQUE INDEX id: String` field on every node. The adapter always queries by `N<Type>({id: value})` (indexed field lookup) rather than by HelixDB's native ID. This is slightly slower (index vs primary key) but preserves referential integrity and enables round-trip migration.

### Schema Examples

**Node — sources:**
```hql
N::Sources {
    UNIQUE INDEX id: String,
    UNIQUE INDEX title: String,
    description: String,
    source_type: String DEFAULT "general",
    start_date: String,
    end_date: String,
    status: String DEFAULT "draft",
    updated_by: String DEFAULT "human",
    last_derived_at: String,
    notes: String,
    created_at: String,
    updated_at: String
}
```

**Edge — bullet_skills:**
```hql
E::BulletSkills {
    From: Bullets,
    To: Skills
}
```

**Edge with properties — bullet_sources:**
```hql
E::BulletSources {
    From: Bullets,
    To: Sources,
    Properties: {
        is_primary: Boolean DEFAULT false
    }
}
```

**Vector — bullet embeddings:**
```hql
V::BulletEmbedding {
    entity_id: String,
    content_hash: String,
    created_at: String
}
```

## 4. Generated Query Catalog

### Naming Convention

`{Operation}{PascalEntityName}` — the adapter maps `(method, entityType)` → query name deterministically. The codegen and adapter share this convention.

### Per-Node Entity (~8 queries × 30 entities = ~240)

| Query Name | Adapter Method | HQL Pattern |
|---|---|---|
| `Add{E}` | `create(type, data)` | `AddN<Type>({...params})` |
| `Get{E}` | `get(type, id)` | `N<Type>({id: id})` |
| `Get{E}By{Field}` | `list(type, {where: {field: val}})` | `N<Type>({field: val})` — one per indexed field |
| `List{E}` | `list(type, {})` | `N<Type>::RANGE(offset, end)::ORDER<dir>(_::{field})` |
| `Count{E}` | `count(type)` | `N<Type>::COUNT` |
| `Update{E}` | `update(type, id, data)` | `N<Type>({id: id})::UPDATE({...params})` |
| `Delete{E}` | `delete(type, id)` | `node <- N<Type>({id: id})` then `DROP node` |
| `ListAll{E}` | `list(type)` (no pagination) | `N<Type>` |

### Per-Edge Entity (~6 queries × 13 junctions = ~78)

| Query Name | Adapter Method | HQL Pattern |
|---|---|---|
| `Add{E}` | `create(type, data)` | `AddE<Type>::From(fromNode)::To(toNode)` |
| `List{E}From` | `list(type, {where: {from_id}})` | `N<FromType>({id: id})::OutE<Type>` |
| `List{E}To` | `list(type, {where: {to_id}})` | `N<ToType>({id: id})::InE<Type>` |
| `Delete{E}From` | `deleteWhere(type, {from_id})` | Traverse + DROP in FOR loop |
| `Delete{E}ByEndpoints` | `deleteWhere(type, {from, to})` | Traverse + filter + DROP |
| `Count{E}From` | `count(type, {from_id})` | `N<FromType>({id: id})::OutE<Type>::COUNT` |

### Named Queries (~5, hand-written)

| Query | SQL Implementation | HQL Advantage |
|---|---|---|
| `traceChain` | 3-table JOIN | `::Out<Derives>::Out<BulletSources>` — native traversal |
| `listBulletsFiltered` | JOIN through junction tables | Traversal + `::INTERSECT` |
| `getResumeWithSections` | 3-level JOIN | `::Out<HasSections>::Out<HasEntries>` |
| `listDriftedBullets` | JOIN + string compare | Traverse + WHERE on snapshot vs current |
| `listDriftedPerspectives` | JOIN + string compare | Same pattern |

### Parameterized Filter Queries (~20)

For operator patterns found in the WhereClause audit:

| Operator | HQL Mapping | Usage Count |
|---|---|---|
| `$in` | `::WHERE(_::{field}::IS_IN(values))` | 4 occurrences |
| `$like` | `::WHERE(_::{field}::CONTAINS(value))` | 3 occurrences (substring only) |
| `$gt` | `::WHERE(_::{field}::GT(value))` | 1 occurrence |

**Total: ~343 pre-compiled queries**

## 5. WhereClause Coverage

### Audit Summary

Full audit of all WhereClause patterns used by services and routes:

| Pattern | Count | Coverage |
|---|---|---|
| Simple equality (`{field: value}`) | ~95 | Tier 1: direct query routing |
| Composite equality (`{f1: v1, f2: v2}`) | ~18 | Tier 1: edge traversal or multi-field lookup |
| `$in` | 4 | Tier 2: parameterized `IS_IN` query |
| `$like` | 3 | Tier 2: parameterized `CONTAINS` query |
| `$gt` | 1 | Tier 2: parameterized `GT` query |
| `$or` | 2 | Tier 3: client-side fallback |
| `$and` | 3 | Tier 3: client-side fallback |

**Not used anywhere:** `$ne`, `$isNull`, `$isNotNull`, `$gte`, `$lte`

### Extension Future Patterns

Browser extension (epic 3bp) upcoming features introduce no new operator patterns:
- JD dedup by URL: simple equality
- Org fuzzy search: `$like` (already covered)
- Answer bank / config / logs: simple equality, write-only

### Tiered Translation Strategy

**Tier 1 (~95% of calls):** Simple equality on indexed fields → route to pre-compiled `Get{E}By{Field}` or edge traversal queries.

**Tier 2 (~4% of calls):** Parameterized filter queries for `$in`, `$gt`, `$like` operators → pre-compiled queries with operator-specific HQL.

**Tier 3 (~1% of calls):** Complex compound `$and`/`$or` with nested operators → `ListAll{E}` + filter in TypeScript. Only 5 occurrences in the codebase (org search, note search).

## 6. HelixAdapter Implementation

### Class Structure

```typescript
class HelixAdapter implements StorageAdapter, GraphCapableAdapter, VectorCapableAdapter {
  readonly capabilities: AdapterCapabilities = {
    graph: true,
    vector: true,
    transactions: false,  // no native transactions; fake transactions satisfy the interface
    nativeCascade: false,  // ELM handles all cascade/restrict/setNull in TypeScript
  }

  private client: HelixDB
  private edgeEntities: Set<string>
  private queryNameMap: Map<string, string>
  private entityMap: EntityMap
}
```

### Constructor Startup

1. Instantiate `new HelixDB(url)` (helix-ts client)
2. Scan entity map — classify entities as node or edge (composite PK + 2+ FK fields = edge)
3. Build query name map — `("create", "bullets")` → `"AddBullets"`

### CRUD Method Routing

```
create(entityType, data)
  ├─ node? → client.query("Add{E}", data)
  └─ edge? → client.query("Add{E}", {fromId, toId, ...properties})

get(entityType, id)
  └─ client.query("Get{E}", {id})

update(entityType, id, data)
  └─ client.query("Update{E}", {id, ...data})

delete(entityType, id)
  └─ client.query("Delete{E}", {id})

list(entityType, opts)
  ├─ no where → "List{E}" with pagination params
  ├─ simple equality → "List{E}By{Field}" or edge traversal
  ├─ $in/$gt/$like → "Filter{E}By{Field}{Op}"
  └─ compound → "ListAll{E}" + filterInTypeScript(rows, where)

count(entityType, where)
  ├─ no where → "Count{E}"
  └─ with where → same routing as list, Count variants

deleteWhere(entityType, where)
  ├─ edge + from_id → "Delete{E}From"
  ├─ edge + endpoints → "Delete{E}ByEndpoints"
  └─ node → list matching IDs, then loop delete

updateWhere(entityType, where, data)
  └─ list matching IDs, then loop update
```

### Fake Transactions

HelixDB uses a single-writer LMDB model with no client-exposed transactions. The adapter implements `Transaction` with buffered writes:

- Reads (`get`, `count`) execute immediately
- Writes (`create`, `update`, `delete`, `deleteWhere`, `updateWhere`) are buffered
- `commit()` flushes the buffer sequentially
- `rollback()` discards un-flushed operations; logs warning for already-flushed ops

This satisfies the `StorageAdapter` interface contract. The ELM's cascade logic already runs in TypeScript and calls individual adapter operations, so atomicity is best-effort.

### Response Normalization

- HelixDB's auto-generated `ID` → mapped to entity's `id` field
- Edge responses → flattened to junction table shape: `{bullet_id: fromId, skill_id: toId, ...properties}`
- ELM handles deserialization (bools, JSON, lazy fields) as usual

### Error Mapping

| HelixDB Error | StorageError |
|---|---|
| 404 / empty result | `notFound(entityType, id)` |
| 409 / unique violation | `uniqueViolation(entityType, field, value)` |
| Connection refused | `connectionError(message)` |
| Other | `adapterError(message)` |

## 7. Multi-File Schema Build System

### Philosophy

Human-friendly multi-file source → machine-friendly single-file output. Both version controlled.

- `src/schemas/*.hx` — hand-maintained, organized by domain
- `src/queries/*.hx` — machine-generated from entity map by `codegen.ts`
- `dist/schema.hx` + `dist/queries.hx` — compiled output consumed by HelixDB

### File Structure

```
packages/core/src/storage/adapters/helix/
├── db/
│   ├── src/
│   │   ├── schemas/
│   │   │   ├── content.hx          (sources, bullets, perspectives)
│   │   │   ├── taxonomy.hx         (skills, domains, archetypes, industries, role_types)
│   │   │   ├── organizations.hx    (orgs, campuses, aliases, tags)
│   │   │   ├── resumes.hx          (resumes, sections, entries, templates)
│   │   │   ├── jobs.hx             (JDs, summaries)
│   │   │   ├── qualifications.hx   (credentials, certifications)
│   │   │   ├── contacts.hx
│   │   │   ├── user.hx             (profile, notes, note_references)
│   │   │   ├── system.hx           (prompt_logs, embeddings, pending_derivations)
│   │   │   └── edges.hx            (all E:: definitions)
│   │   └── queries/
│   │       ├── content.hx          (mirrors schema grouping)
│   │       ├── taxonomy.hx
│   │       ├── ...
│   │       └── named.hx            (hand-written: traceChain, etc.)
│   │
│   ├── dist/
│   │   ├── schema.hx               (concat of src/schemas/*)
│   │   └── queries.hx              (concat of src/queries/*)
│   │
│   └── Dockerfile                   (HelixDB build: install CLI, copy dist/, helix build, run)
│
├── codegen.ts                       (entity map → src/queries/*.hx)
├── build.ts                         (src/ → dist/ compile; dist/ → src/ extract)
├── helix-adapter.ts
├── helix-transaction.ts
├── helix-where.ts
├── helix-response.ts
├── helix-named-queries.ts
└── __tests__/
    ├── helix-adapter.test.ts
    ├── helix-where.test.ts
    ├── codegen.test.ts
    └── build.test.ts
```

### Build Commands

```
just helix-gen       codegen.ts: entity map → src/queries/*.hx
just helix-build     build.ts: src/**/*.hx → dist/schema.hx + dist/queries.hx
just helix-extract   build.ts: dist/*.hx → src/**/*.hx (reverse)
just helix-push      helix push dev (or docker compose up helixdb --build)
just helix-dev       helix-gen + helix-build + helix-push (all-in-one)
```

### Round-Trip Guarantee

The build script adds section markers (`// === content.hx ===`) so extract can split deterministically. Invariant: `extract(build(src)) === src`.

## 8. Two-Way Migration

### Entity Ordering

Entities migrated in dependency order (parents before children, nodes before edges):

| Phase | Entities |
|---|---|
| 1 — Leaf nodes | `skill_categories`, `domains`, `archetypes`, `industries`, `role_types`, `user_profile`, `resume_templates` |
| 2 — Core | `organizations`, `skills`, `contacts`, `summaries` |
| 3 — Content | `sources`, `job_descriptions`, `credentials`, `certifications`, `prompt_logs`, `user_notes` |
| 4 — Source subtypes | `source_roles`, `source_projects`, `source_education`, `source_presentations` |
| 5 — Content children | `bullets`, `org_campuses`, `org_aliases`, `org_tags` |
| 6 — Derived | `perspectives`, `resumes`, `embeddings`, `pending_derivations` |
| 7 — Resume structure | `resume_sections` → `resume_entries`, `resume_skills`, `resume_certifications` |
| 8 — Junctions (edges) | All 13 junction tables |

### Migration Runner

```typescript
interface MigrationOptions {
  source: StorageAdapter
  target: StorageAdapter
  entityMap: EntityMap
  onProgress?: (entity: string, count: number, total: number) => void
  dryRun?: boolean
}

async function migrateAll(opts: MigrationOptions): Promise<MigrationReport>
```

Per entity: `source.list()` → transform if edge → `target.create()` per row → count verification.

### Verification Pass

| Check | Method |
|---|---|
| Entity counts match | `count()` per entity on both adapters |
| Content hashes match | SHA-256 of sorted row JSON for sampled entities |
| FK consistency | For each edge, verify both endpoint nodes exist |
| Named query parity | Run each named query on both, compare result shapes |

### Just Targets

```
just helix-migrate-to      sqlite → helix
just helix-migrate-from    helix → sqlite
just helix-verify          verification pass on both backends
```

## 9. Docker Compose & Dev Workflow

### HelixDB Service

```yaml
services:
  helixdb:
    build:
      context: packages/core/src/storage/adapters/helix/db
      dockerfile: Dockerfile
    ports:
      - "6969:6969"
    volumes:
      - helix-data:/data
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:6969/health"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  helix-data:
```

The Dockerfile installs the HelixDB CLI, copies `dist/*.hx`, runs `helix build` to compile HQL → Rust, and runs the resulting binary.

### Adapter Selection

```bash
FORGE_STORAGE=sqlite   # default
FORGE_STORAGE=helix    # use HelixAdapter
HELIX_URL=http://localhost:6969
```

### Test Strategy

| Type | Runs Against | Gate |
|---|---|---|
| Unit (where routing, response normalization, codegen) | Mocked client | Always — `just test` |
| Codegen (generated .hx validity, naming convention) | File system | Always — `just test` |
| Build (round-trip src ↔ dist) | File system | Always — `just test` |
| Integration (full CRUD) | Live HelixDB | `FORGE_STORAGE=helix` |
| Migration (round-trip sqlite ↔ helix) | Both backends | `FORGE_STORAGE=helix` |
| Named query parity | Both adapters | `FORGE_STORAGE=helix` |

## 10. What This Phase Does NOT Change

- `StorageAdapter` interface (`adapter.ts`) — unchanged
- `EntityLifecycleManager` (`lifecycle-manager.ts`) — unchanged
- Entity map (`entity-map.data.ts`) — unchanged
- Any service files — unchanged
- SQLite adapter — unchanged
- Any route files — unchanged

The HelixAdapter is purely additive. It implements the existing interface alongside the SQLite adapter.
